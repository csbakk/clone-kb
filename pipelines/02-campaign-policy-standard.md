# 캠페인 운영 정책 표준 (`_POLICY.md` 템플릿)

> **오너 채택(2026-07-20): 이 규칙을 클론 캠페인의 정식 표준으로 삼는다.** notion 캠페인에서 실전 검증된 `_POLICY.md`를 일반화한 것. 새 캠페인은 **리포 루트에 `_POLICY.md`를 만들고 아래를 복사**한 뒤 프로젝트값(포트·경로·게이트명)만 채운다.
>
> **왜 파일인가**: 긴 세션에서 정책은 오케스트레이터 컨텍스트에서 유실된다(과거 행동 습관이 새 지시를 이긴다). 워커는 매번 새 컨텍스트이므로 **"작업 전 `_POLICY.md`를 읽어라"를 브리프 1번 지시로** 두면 항상 현행 규칙이 적용된다. 정책 변경 시 **파일을 먼저 고치고(변경로그 한 줄) 가동 중 워커에 주입**한다. 브리프에 정책을 복붙하지 마라(템플릿 오염).

---

## 템플릿 (복사해서 쓰기)

### ★★★ §0. 최상위 도크트린 — 모든 요소는 실물 1:1
**어떤 요소를 만들거나 고치든(신규·버그 수복·placeholder·상태·빈 화면 전부 예외 없음) "대충 화면 흉내"·근사·합성 데모로 끝내지 마라. 반드시 원본 앱에서 최하단 자식부터 실측해 1:1 복사한다:**
1. **골격/구조** — 최하단 자식 → 콘텐츠 루트/body까지 층별 tag·class·role·display·position·층수(chain)
2. **스타일** — 각 층의 fs/fw/lh·color·bg·margin/padding 4방·border·radius 전 속성(computed 실측, 추측 금지)
3. **상태** — 평상시 + hover + focus + pressed + 진입/전환 후 변화(state matrix)
4. **애니메이션/타이밍** — duration·easing·프레임 지문

- **오너 리포트 버그는 실제 UI 경로(사용자가 하는 조작 그대로)로 재현**하고, 수복 후 원본↔클론 비교 시각 증거로 증명한다. **합성 주입만으로 "정상/재현 안 됨" 판정 금지.**
- **오너 조작 영상은 메타데이터부터** — 캡처 도구가 조작기록(클릭좌표·키·타이밍)을 영상에 내장한다면 프레임 눈대중 전에 그것부터 읽어 조작을 정본으로 확정한다(notion: `ffprobe -show_entries format_tags=capdraw_timeline|capdraw_events`, 사이드카 `.events.json`/`.timeline.md`도 동일).
- **오케는 이 원칙을 매 브리프에 재주입하지 않는다.** 워커가 여기서 읽고 무조건 적용. 애매하면 대충 넘기지 말고 실측하거나 티켓으로 정직 보류.

### §1. 검증 (최소주의)
- 워커: **빌드/타입체크 + 스모크 + 변경 표면의 타깃 게이트만.** 전판 금지.
- 오케: **이중 재검증 금지** — 워커의 수치 보고를 신뢰하고 바로 push. 스팟 재검증은 ①데이터 손실 위험 ②수치 없는 보고 ③신규 패턴일 때만.
- 전판·전수 게이트: **오너 요청 또는 사전 고지된 무인 런 경계에서만.**
- 게이트 FAIL: 단독 재실행으로 플레이크/실물 판정. 실측 근거 없는 값 변경 금지.

### §2. 브라우저 (→ 상세 [[techniques.non-intrusive-browser-automation]])
- 클론 작업 = 전용 브라우저(예: 번들 크로뮴 CDP 포트). 원본 = 로그인 세션 있는 창의 **기존 탭 재사용만**.
- **원본 탭 churn 금지**(봇차단 위험): 재로그인·탭 재생성 반복 금지. **로그인 화면이 보이면 어떤 시도도 없이 중단·보고.**
- **중단 사유 판별**: 중단은 **로그인 화면 또는 CDP 무응답(attach/evaluate hang)일 때만.** 탭 개수 변화·낯선 탭은 오너가 같은 브라우저를 쓰는 흔적일 수 있으니 계속 진행(애매하면 오케에 판정 요청).
- **네비게이션**: `domcontentloaded`+수동대기(`wait_until="load"`는 hang 트리거 전력), 간격 2초+, **세션당 goto 예산 명시**(기본 15회).
- 원본 **기존 문서 = read-only**(hover·토글 열닫·팝오버+Escape 등 가역 조작만). **새로 만든 스크래치 페이지에서는 생성/수정/파괴 허용**(실측 실험장, 끝나면 아카이브).
- **★포커스 탈취 금지** — 오너와 같은 기기를 쓰므로 창이 전면으로 뜨면 키입력이 꼬인다. `bring_to_front()`·osascript `activate`·`window.focus()` **금지**. 브라우저 작업은 백그라운드 CDP로: 계측=`Runtime.evaluate`, 키/클릭=`Input.dispatchKeyEvent`/`dispatchMouseEvent`, 캡처=`Page.captureScreenshot`. OS 레벨 포커스가 꼭 필요하면(trusted 키입력·클립보드) **사전 승인 + 최소 점유 후 반환**.
- **★새 탭도 포커스를 뺏는다** — `new_page()` 금지, **기존 탭 재사용**이 원칙. 정말 필요하면 CDP `Target.createTarget(background: true)`, 끝나면 `Target.closeTarget`으로 정리(잔재 탭이 쌓이면 다음 세션의 탭 선택이 오염된다).
- **★역할 분리 — 계측=headless / 픽셀=headful**: 계측·동작 재현·게이트·버그 재현은 headless(포커스 무침해, 렌더 엔진 동일). **픽셀 비교 캡처만 headful 기존 창** — 원본과 동일 렌더 조건이어야 유효하고 headless는 폰트 서브픽셀이 달라 수치가 오염된다.

### §3. 실행 규율
- **오케=오케스트레이션만**(브리프·push·보고). 실작업은 하위 모델 워커. 워커는 **이번 턴 완결**(백그라운드 던지고 턴 종료 금지, 포그라운드 실행).
- 커밋은 **자기 파일만 정밀 스테이징**(`git add -A` 금지 — 병렬 워커 충돌). push는 오케만.
- **보고 HTML 자동 open 금지** — 최종 보고 시 1회만.
- **★비교/검증 시각 증거 필수**: 검수·버그·파리티 보고는 **원본↔클론 비교 시각 증거 첨부** — 정적=나란히 캡처(동일 영역·동배율), 동작=영상. 텍스트·수치만으로 "해결/일치" 보고 금지. 수복 후 양쪽 재캡처를 보고서가 임베드.
- 수복은 **실측 근거만**(추측 px/추측 동작 금지). 확정 못 하면 티켓으로 정직 보류.
- **★오너 리포트 버그는 실경로 재현**: 합성 데이터와 실제 앱 경로는 데이터 형태가 달라 결과가 갈린다. 오너가 준 조작 순서를 그대로 재현하고, 회귀 게이트에도 실경로 케이스를 포함.
- **구조-우선 도크트린**: 골격(체인 전체)→스타일→상태→동작. 픽셀 지표는 구조 게이트와 **쌍으로만** 신뢰([[techniques.structure-first-cloning]]).

### §4. 변경로그
정책을 고칠 때마다 **날짜 + 한 줄 + 배경(왜)**. 배경이 있어야 나중에 규칙을 되돌릴지 판단할 수 있다.

## 각 규칙이 생긴 이유 (notion 실증 — 같은 값을 두 번 치르지 마라)

| 규칙 | 계기 |
|---|---|
| §0 실물 1:1 최상위 | 오너가 "최하단부터 골격·CSS 다 가져오라"를 내내 반복 지시했는데 워커가 대충 흉내로 빠져 매번 재지시하게 만든 시스템 결함 |
| 합성 판정 금지 | 빈 토글 placeholder를 합성 주입에서 "정상"이라 오판 → 오너 재보고 2회 (실제론 **중첩** 경로에서만 깨졌음) |
| 영상 메타 우선 | 캡처 도구가 조작기록을 내장 → 프레임 눈대중보다 정확, 재현 실패를 재현 성공으로 바꿈 |
| 시각 증거 필수 | 텍스트 주장만으론 오너가 눈으로 검증 불가 |
| 중단 사유 판별 | 탭 변화를 데드락으로 오판해 워커 2명이 과잉 중단 |
| 포커스/새 탭 금지 | 오너가 같은 기기에서 작업 중인데 창이 튀어나와 키입력이 꼬임(3회 호소) |
| 계측 headless/픽셀 headful | headless 전환이 포커스 문제를 없애지만 픽셀 비교의 "동일조건" 원칙을 깨뜨림 → 목적별 분리 |
| 정밀 스테이징 | 병렬 워커가 서로의 미커밋 파일을 쓸어담는 사고 |
| 검증 최소주의 | 워커마다 전판 게이트를 돌려 무관 게이트가 90%였음(오너 3회 지적) |

## 도입 절차

1. 새 캠페인 킥오프 시 [[pipelines.00-campaign-kickoff-playbook]] §5 체크리스트대로 **리포 루트에 `_POLICY.md` 생성** → 이 템플릿 복사.
2. 프로젝트값 채우기: 포트(클론/원본 CDP·dev 서버), 스크래치 루트, 게이트 이름, push 계정.
3. **모든 워커 브리프의 1번 지시 = "먼저 `_POLICY.md`를 읽어라."**
4. 정책 변경 시: 파일 먼저 수정(+변경로그) → 가동 중 워커에 주입 → 메모리/clone-kb는 이 파일을 가리키는 포인터로만 유지.
