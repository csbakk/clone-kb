# 안전장치 설계 원칙 — 가용성을 죽이는 보호는 보호가 아니다

**한 줄**: 데이터 유실을 막으려고 넣은 보호 장치가 **회복 경로 없이·전체 원자적으로** 작동하면, 막으려던 사고보다 더 나쁜 "전면 가용성 상실"을 만든다. 기본값은 **fail-open + 시끄러운 로그**로, 엄격 모드는 옵트인 플래그로 둔다. 진짜 안전망은 **세대 백업**이다.

## 왜 (notion 캠페인 실증 — 가장 값비싼 교훈)

데이터 유실 사고(헤드리스 워커가 실사용 페이지를 열고 뮤테이션을 실행해 서브트리 블록이 즉시 소멸) 후, 재발 방지로 **rev 기반 경합 감지**(클라이언트 `baseRev`가 서버 rev와 다르면 409로 거부)를 넣었다. 그런데:

1. **클라이언트의 409 회복 경로를 실증하지 않고** 서버 거부부터 켰다 — 클라이언트가 409를 받아도 `baseRev`를 최신으로 재동기화하지 못해 그 자리에 멈췄다.
2. **배치 전체를 원자적으로 거부**하게 만들었다 — 경합과 무관한 페이지들의 쓰기까지 같이 막혔다.

결과: 클라이언트는 계속 낡은 `baseRev`로 재시도 → 서버는 계속 409 → **무한 409 루프**(로그 1031회) → **모든 저장이 조용히 실패**. 오너는 편집이 저장되는 줄 알았지만 실제로는 아무것도 반영되지 않고 있었다.

## 핵심 원칙

1. **가용성을 죽이는 보호는 보호가 아니다.** 일부 손실(last-write-wins 등)보다 **전면 실패**(아무것도 저장 안 됨, 그것도 조용히)가 명백히 더 나쁠 수 있다. "더 안전해 보이는" 메커니즘이 실제로는 더 위험한 실패 모드를 여는지 먼저 따져라.
2. **회복 경로를 먼저 실증하고 나서 강제하라.** 거부/차단 로직을 켜기 전에, 거부당한 클라이언트가 스스로 복구하는 경로(재동기화·재시도·백오프)가 **실제로 작동하는지 검증**한 다음에 강제를 켠다. 순서를 바꾸면(강제 먼저, 회복은 나중) 이번처럼 무한 루프에 빠진다.
3. **기본값은 fail-open + 시끄러운 로그.** 감지·기록은 항상 하되, 거부는 옵트인 엄격 모드(예: 환경변수 플래그)로만 켠다. 경합 자체를 막는 게 아니라 **경합을 알아채고 기록하는 것**이 기본 책임이다.
4. **세대 백업이 있으면 대부분의 보호를 fail-open으로 둘 수 있다.** 진짜 안전망은 실시간 거부 로직이 아니라 **쓰기 직전 세대 백업**(N세대 보관 + 크기 급감 시 경고)이다 — 값싸고 확실하고, 클라이언트 로직의 버그에 영향받지 않는다. 이 원칙의 사고 당시엔 세대 백업이 없어 복구가 우연(diff 재구성)에 의존했다 — 이후 배포됨.

## 짝 규칙 — 쓰기 직전 세대 백업
- 매 쓰기 직전 이전 상태를 `<key>-backup-N`류로 스냅샷, N세대 보관.
- **블록 수/크기가 급감**하면(예: 이전 대비 대폭 축소) 경고 로그 — 데이터 유실의 조기 신호.
- [[techniques.persist-migration-safety-net]]의 "변환전 백업"과 같은 계열이지만, 이건 **스키마 변경이 아니라 매 쓰기**에 적용되는 상시 안전망이라는 점이 다르다.

## ★4. 동기화는 양방향으로 검증하라

실시간 동기화(원격 patch를 받아 즉시 반영)를 붙일 때 "주입한 내용이 새로고침 없이 화면에 뜨는가"만 확인하고 켰다가, **반대 방향**(로컬의 아직 저장 안 된 변경이 뒤늦게 도착한 오래된 원격 패치에 덮이는 경우)을 놓쳤다. 실제 사고: 토글을 펼친 직후 write-through 디바운스 창(예: 800ms) 안에서 원격 patch가 도착하면, **더 새로운 로컬 상태가 더 오래된 파일 상태로 덮여** 오너의 편집이 되감겼다.

- **교훈**: "넣는 것"만 검증하고 "잃는 것"을 안 보면 반드시 샌다. 실시간/양방향 동기화가 있는 기능은 **양쪽 방향을 각각 별개의 회귀 케이스로** 다뤄야 한다.
- **회귀 테스트에 반드시 포함**: "디바운스 창 안에서 원격 patch가 도착해도, 아직 저장되지 않은 로컬 변경(dirty 상태)이 살아남는가?" — 이 케이스가 없으면 위 사고는 재발한다.
- **구현 힌트(실증)**: 원격 patch 적용 전 해당 문서/블록이 dirty(미저장 로컬 변경, 포커스 여부와 무관 — 토글·체크박스 조작도 dirty로 취급)면 적용을 보류하고, 로컬 변경을 먼저 flush한 뒤 재평가한다.
- 이 사고는 위 rev 경합 사고와 **같은 날, 같은 계열의 실시간 동기화 기능에서 함께 발생**했다 — 신기능 하나가 여러 실패 축(가용성·정합성)을 동시에 열 수 있다는 신호이기도 하다.

## 함정
- "회귀 감지 로직을 추가했다" ≠ "안전해졌다". 로직이 **틀린 조건에서 거부하면 오히려 위험**하다 — 사고 전엔 무보호(일부 손실 가능), 사고 후엔 전면 무저장(전부 손실 가능)이라는 더 나쁜 상태로 이동했다.
- "실시간 동기화 켜짐 = 편집이 실시간으로 반영됨"이라는 확인만으로 출시하지 마라. 되감김(rewind)은 정반대 방향의 실패라 순방향 테스트로는 절대 안 잡힌다.
