# 구조-우선 클론 (Structure-First)

**한 줄**: 클론의 복사 순서는 **①골격(DOM 구조) → ②스타일(computed) → ③동작(이벤트·JS)** — 골격을 먼저 복사해야 스타일이 1:1 이식되고 픽셀이 "보정 없이" 맞는다. **오너 직접 채택 원칙(2026-07-17), 모든 클론 캠페인의 1순위.**

## 왜 (notion 캠페인 실증)

스타일-먼저(값 포팅)로 가면 **보정 스택**이 생긴다: 요소 A의 8px 차이를 요소 B의 마진으로 상쇄 → 특정 상태에선 픽셀이 맞지만 내용이 바뀌면 깨진다(번호목록: 마커와 텍스트가 다른 부모라 높이 어긋남). 수렴 루프 이터레이션 7회의 갭 대부분이 구조 기원이었다 — 골격이 같았으면 스타일 단계가 거의 공짜였다.

## 순서 원칙

1. **골격**: 블록/컴포넌트 단위로 실물의 DOM 스켈레톤을 채택 — 태그(h1 vs input), a11y role(textbox vs heading), **부모-자식 그룹핑**(마커+텍스트 단일 부모), 래퍼 체인, contenteditable 배치. 실물 스켈레톤 자동 추출(태그·role·padding·그룹핑 덤프) → 클론 렌더러가 같은 골격을 뱉게 → **구조 게이트**로 동치 고정.
2. **스타일**: 골격이 같으면 computed 값·셀렉터 규칙을 거의 그대로 이식([[techniques.measured-css-porting]]) — 보정 불필요.
3. **동작**: 이벤트·키보드·상태 전이·JS 로직 복사/재현. 검증은 [[techniques.pixel-diff-baseline]](픽셀)+구조 게이트(골격)+기능 게이트(동작) 3축.

→ **복사 가능한 것을 전부 복사하면 가시·상호작용 층은 99% 도달 가능.** 잔여 1% = 복사 불가능물(서버 로직·외부 iframe 콘텐츠·백엔드 데이터)로, 재구현 대상.

## 함정

- **스타일부터 시작한 것은 큰 실수였다(오너 판정)**: 픽셀 91→96%로 수렴했는데도 오너 DevTools 인스펙션 결과 **골격은 사실상 전부 불일치**(제목 태그/role/패딩축·리스트 그룹핑·거터·보더 층·래퍼 체인). 픽셀 지표가 보정 스택의 상쇄를 "일치"로 오보고한 것 — **골격 게이트 없는 픽셀 지표는 거짓 안심**. 7이터레이션+전면 retrofit 비용. notion에서 지불하고 배운 것.
- 골격 리팩터는 렌더러 심장부를 건드림 — 반드시 게이트망([[techniques.regression-harness-suite]]) 선행 + 블록 타입 단위로 점진.
- 이미 스타일-먼저로 진행된 캠페인은 골격을 **retrofit**(블록 단위 골격 교체, 픽셀 기준선으로 무회귀 확인)한다.
- ★**"무해 단순화" 판정은 잠정이다 — 오너 확인 전까지 확정 아님.** canvas 세션22(2026-07-17) 스켈레톤 매트릭스가 체인 층수 격차(real13 vs clone10)·상태전이 메커니즘 격차(mount/unmount+reflow vs opacity토글)를 "최종 픽셀 등가라 무해"로 자체 판정했으나, 오너가 결산 시점에 개입해 뒤집었다("프로들이 그 층 구조를 이유있게 만들었고, 층까지 맞춰야 실물이 기능을 추가해도 클론이 그 구조를 따라갈 수 있다"). **빌더·검증자가 픽셀 등가를 근거로 골격/메커니즘 차이를 스스로 "무해"라 닫지 말 것** — 저위험 잔여로 문서화하되 오너 승인 전까지는 열린 항목으로 취급.

## 오너 결정 — 체인깊이 A/B → B 채택(전층 일치, total-fidelity)

2026-07-17(canvas 세션22 결산) 오너가 원칙을 확장: 골격 순서("①골격→②스타일→③동작")뿐 아니라 **조상체인 층수·상태전이 메커니즘까지 실물과 동치화하는 것을 목표**로 삼는다("전층 계약 클론"). A안(실용 — 최종 픽셀 등가면 층수/메커니즘 차이 수용)은 기각, B안(충실 — 층수까지 완전 일치, 비용 감수)을 채택. 근거는 "구조가 이유 없이 존재하지 않는다 + 실물의 향후 기능 추가를 클론이 같은 구조로 따라갈 수 있어야 한다"는 유지보수성 논리 — 단발 픽셀 파리티가 아니라 **구조적 추종 가능성**이 목표. 상세: canvas `clone-kb runs/2026-07-17-canvas-remaining-parity.md` §5, `status/canvas.md` 세션22 헤더.

## 실증 2 — canvas retrofit (2026-07-17, 세션21)

notion은 처음부터 구조-우선으로 설계된 캠페인이었지만, canvas는 세션1~20을 스타일-먼저(값 포팅) 방식으로 누적 진행해온 캠페인이다 — 위 §함정의 "이미 스타일-먼저로 진행된 캠페인" 시나리오를 실제로 만난 첫 사례. 소급 절차:

1. **정찰 우선, 신규 캡처 없이**: 기존 RIP 델타(19상태) + 서술형 RECON 문서 + 현재 소스코드(git log/grep) 3축 교차검증만으로 확정 갭 3건(헤더 span래퍼·라이트박스 메타패널·프롬프트 골격)을 추려냄. 델타 문서의 스테일 항목(이미 다른 커밋으로 해결된 5건)과 캡처 아티팩트(소스가 이미 자인한 하네스 오탐)를 먼저 걸러낸 뒤에만 "진짜 구조 갭"으로 확정 — 신규 실측 없이 기존 산출물 재해석만으로 우선순위를 세울 수 있음을 보임.
2. **블록 단위 점진 리트로핏**: 헤더span(1줄, 3곳) → 라이트박스 메타패널(신규 섹션 추가, 기존 구조 안건드림) → 프롬프트 골격(렌더러 심장부, 6개 소비처 순차 확산) 순으로 리스크가 낮은 것부터. 매 블록마다 RIP isolated resweep으로 무회귀 확인 후 다음 블록 진행.
3. **결과**: 헤더span 1건이 6상태 diff -40%, 라이트박스 1건이 -29% 동반 감소 — notion과 동일하게 "구조 정합이 스타일 레이어를 공짜로 데려온다"는 메커니즘이 재현됨. 프롬프트 골격은 이미 프로덕션 검증된 contentEditable 패턴(다른 노드 타입의 `useEditableSeed`)을 재사용해 렌더러 심장부급 리팩터도 신규 리스크 없이 6타입 확산 가능함을 보임(매 라운드 이전 타입 전체 무회귀 재확인 18/18×6회).

→ retrofit이 스타일-먼저 캠페인에도 유효한 탈출 경로임을 확인. 상세: `clone-kb/runs/2026-07-17-canvas-structfirst.md`, canvas repo `ref/_STRUCTURE_FIRST_ROADMAP.md`.

## 확장 결정 — 셸까지 1:1 (B안, 오너 채택 2026-07-17)

골격 복사 범위는 **잎→본문루트뿐 아니라 앱 셸(body까지) 체인 전체 1:1**. 근거(오너): ①원저작 레이어 구조엔 엔지니어링 이유가 인코딩돼 있어(스크롤/이벤트 위임/가상화 경계) 구조를 복사하면 이유의 효과까지 따라옴 ②추후 원본에 기능이 추가될 때 "아는 층"에 꽂히므로 추적 비용 급감. 셸은 전 블록 공통 조상이라 **1회 추출**이면 됨(블록별 재조사 불요).

## 전층 클론 지도 (화면 복사를 넘어 — 2026-07-17)

①DOM 골격(셸 포함) → ②스타일(computed) → ③상태 변화(hover/press/focus 매트릭스: 상태별 DOM 마운트+computed 델타) → **④동작 골격 = 이벤트 리스너 지도**(CDP getEventListeners — "어느 층이 무엇을 듣나"를 JS 재현 전에 복사, 구조-우선의 동작층 재귀 적용) → ⑤애니메이션 지문([[techniques.animation-ripper]]) → ⑥타이밍 지문(디바운스·오토세이브 주기·낙관적 업데이트 순서) → ⑦네트워크 계약(공개 API 파리티) → ⑧접근성 트리 전수 diff → ⑨엣지 상태(빈/로딩/에러). 잔여 비복사물=서버 로직·외부 콘텐츠만.

## ★측정 기준을 먼저 실물과 동치화하라 — diff 필드 전수 + 최하단 자식 대조 (notion 2026-07-19)

구조-우선의 성패는 **"무엇을 갭으로 세느냐"(비교 기준)에 달려 있다.** 골격 게이트가 있어도 그 게이트가 비교하는 축이 실물과 다르면, 안 맞는 것을 맞다고 보고한다 — "골격 게이트 없는 픽셀은 거짓 안심"의 심화판이다.

1. **비교 필드 전수화**: 자동 diff는 `font-size/weight/line-height·margin·padding·border·radius·display·color/bg`를 **전부** 비교하고 하나라도 다르면 GAP. **골격 깊이(chain_depth = 잎→루트 층수)도 별도 GAP 조건.** "폰트는 디자인 토큰이라 달라도 됨" 같은 완화 전제를 두지 마라 — 클론 목표는 실물과 동일이다. notion에서 이 완화 전제 하나가 OK 16/17을 만들었고, 전수 비교로 바꾸니 전 타입이 GAP으로 드러났다.
2. **최하단 자식부터(nested-context 대조)**: 타입당 top-level 대표 1개만 격리 렌더해 비교하면, 실물에서 항상 중첩돼 나타나는 블록(리스트·todo·quote·file)의 골격 갭을 코퍼스에 top-level 표본이 없다는 이유로 놓치거나 "아티팩트"로 오판한다. **실물 리프의 조상 구조(중첩 N단·컬럼 안·토글 안)를 클론에도 동일 재현**해 같은 컨텍스트의 최하단 자식끼리 대조해야 진짜 골격 갭이 확정된다.
3. **"구조를 바꿨는데 픽셀 무하락" = 정합의 증거**: 골격을 실물 모델로 교체했을 때 픽셀이 유지되거나 오르면(notion `.blk` flex→block 전환 시 풀블록 +3.1pp) 그 골격이 옳다는 뜻이다. 픽셀이 깨지면 실물 모델을 잘못 이해한 것 → 롤백·재실측. 이 무하락 A/B가 대수술의 안전장치다.
4. **보정 스택은 골격이 틀렸다는 신호**: 리프 base(`.blk`)에 flex+padding을 깔고 그 위에 타입별 margin-top 26/22px 보정을 쌓아 픽셀만 맞춘 상태가 바로 오너가 말한 "computed 복사 최대 실수"의 실체. 보정 상수가 쌓여 있으면 그 아래 골격을 의심하라.

→ **99%의 조건**: (a)비교 기준을 실물과 동일 축으로 세우고(전 필드+골격깊이+최하단 자식) (b)리프를 전수 측정해 수렴을 확인하고(부분 커버리지는 "안 본 갭"을 숨긴다) (c)동작/상태/엣지 축은 골격과 별개 트랙으로 함께 닫아야 한다. 셋 중 하나라도 빠지면 "높은 지표"는 거짓 수렴일 수 있다.

## ★적용 범위 caveat — 대상 노드가 DOM일 때만 전제 (canvas 2026-07-18 WebGL 발견)

골격-우선(structure-first) 전체가 **암묵적으로 "복사할 DOM이 존재한다"를 전제**한다. canvas 캠페인이 2026-07-18에 실물 Higgsfield Canvas가 노드를 **DOM이 아니라 WebGL `<canvas>` 픽셀로 렌더**하도록 업그레이드된 것을 발견했다 — `.react-flow__nodes` children 0, 노드 텍스트가 `document.body.innerText`에 부재, 뷰포트 전체 WebGL `<canvas>` 존재 3-신호로 확정([[techniques.webgl-node-rendering-clone]]).

이 경우 **노드 골격 자체가 없으므로** 위 ①~⑨ 전층 지도를 노드 층에 적용할 방법이 없다 — ①DOM 골격부터 복사 대상이 존재하지 않는다. **이는 원칙의 은퇴가 아니라 적용 범위의 경계다**: 셸/툴바/인스펙터/메뉴 등 여전히 DOM인 부분엔 구조-우선이 그대로 유효하고, 오직 캔버스 위 노드 렌더 층만 예외다.

**대체 경로**: 노드 층은 data-model(클립보드/localStorage JSON) + visual(스크린샷) + behavior(좌표 클릭 인터랙션)로 캡처·재현한다 — [[techniques.webgl-node-rendering-clone]] 참고. 클론 자체는 DOM/@xyflow를 유지하기로 오너가 결정(대규모 성능 이득이 없는 규모에서 WebGL 전환 대가가 방법론 핵심을 침해하기 때문) — 노드 렌더 층에 한해 구조-우선을 유예한 사례이지, 원칙을 뒤집은 것이 아니다.

## ★실측이 과거 recon 문서를 상위한다 — 구조는 버전마다 드리프트한다 (canvas 2026-07-19)

골격-우선 전면 적용 중 모델 피커 플라이아웃이 클론(구 07-12 recon 기준 w344·maxH460)과 실물(현재 w400·maxH640)이 크게 어긋난 걸 발견 — **원인은 클론 버그가 아니라 실물이 그 사이 리스타일된 것**. 규칙: **과거 캡처 문서와 라이브 측정이 불일치하면 라이브가 이긴다.** recon 문서는 그 시점의 스냅샷일 뿐, 골격 정합의 기준은 항상 `measure_chain` 실시간 덤프다. 이것이 [[techniques.baseline-then-diff]]의 diff가 노리는 바 — 한 번 채운 baseline을 진리로 굳히지 말고, 매 정합 시 대상 요소는 재실측해 문서를 갱신(이번엔 aspect 리스트에서 실물에 있으나 클론에 없던 `21:9`도 이때 드러남). "체인이 안 맞음"의 첫 감별진단은 **클론 오류 vs 실물 드리프트** 둘 다 열어두는 것.

## ★라이브 테스트 해저드 — HMR이 모듈 카운터를 리셋해 데이터 유실 (canvas 2026-07-19)

골격 수정→CDP로 클론 열어 신요소 배치·확인하는 라이브 검증 루프에서 **기존 노드 1개가 조용히 덮여 유실**되는 사고. 근본원인: `CanvasApp.tsx`의 **모듈-레벨 `idSeq` 카운터가 소스 편집 시 HMR로 1로 초기화** → 이후 추가 노드가 기존 ID와 충돌해 overwrite. 페이지 새로고침 시엔 로드 로직이 재동기화하므로 **배포엔 무영향, 개발 라이브테스트에만 발현**. 교훈 둘: (1)클론 검증 세션에서 **소스를 편집한 뒤엔 배치 테스트 전 새로고침**으로 모듈 상태를 재싱크하라(HMR 중간상태에서 mutate 금지). (2)이 사고 자체가 클론의 잠재버그를 드러냄 — **ID를 모듈 카운터가 아니라 기존 노드 max id 기반으로 유도**하면 HMR·리로드 양쪽에서 충돌이 근본 제거. 라이브테스트 중 정체불명의 상태 오염이 보이면 HMR-리셋되는 모듈 전역을 첫 용의자로.
